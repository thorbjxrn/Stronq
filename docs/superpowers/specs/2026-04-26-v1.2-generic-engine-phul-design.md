# Stronq v1.2: Generic Engine + PHUL

## Context

Stronq v1.0 ships with the DeLorme progressive resistance method — a single, hardcoded program. The engine (DeLormeEngine), view model (WorkoutViewModel), day types (DayType enum), and workout UI (WorkoutSessionView) are all tightly coupled to DeLorme's specific rules: 50/75/100% pyramid sets, Heavy-Light-Medium weekly structure, 5-rep scheme, 7-week cycles, "match Monday" volume logic.

v1.2 refactors the engine into a generic, data-driven system and ships PHUL (Power Hypertrophy Upper Lower) as the second program, proving the abstraction works. This lays the foundation for adding more programs cheaply (v1.3+) and eventually exposing the system to users for custom programs (v2.0).

## Roadmap Overview

| Version | Content | Engine work |
|---------|---------|-------------|
| v1.0 | DeLorme (Classic, Yoked) | Hardcoded |
| **v1.2** | **Generic engine + PHUL** | **Full refactor** |
| v1.3 | nSuns + Candito | Config only |
| v1.4 | GZCL + PPL + AdMob | Config only |
| v2.0 | Custom programs UI | Expose config to users |

After v1.2, adding a program is just defining a new ProgramDefinition — no engine changes needed.

## Core Abstraction: ProgramDefinition

The central data structure that describes any program's rules. DeLorme, PHUL, nSuns, etc. are all just different ProgramDefinition values.

### Structure

```
ProgramDefinition
├── id: String
├── name: String                    ("DeLorme Classic", "PHUL")
├── description: String
├── isPremium: Bool
├── cycleLength: Int?               (7 for DeLorme, nil for repeating)
├── repeating: Bool                 (false = fixed cycle, true = loops indefinitely)
├── introCycle: IntroCycle?         (optional ramp-up weeks)
├── days: [DayDefinition]
```

**Validation:** `cycleLength` and `repeating` are coupled — `repeating: true` requires `cycleLength: nil`, and `cycleLength: nil` requires `repeating: true`. Invalid combinations are programmer errors caught at definition time, not runtime.

### IntroCycle

Optional ramp-up period before the main program begins. Used by DeLorme (2 intro weeks at reduced volume); PHUL does not use one.

```
IntroCycle
├── weeks: Int                      (number of intro weeks, e.g. 2)
├── volumeMultiplier: Double        (fraction of normal volume, e.g. 0.6)
```

During intro weeks, `repeatCount` for all set groups is scaled by `volumeMultiplier` (rounded up). Intensity percentages and rep targets stay the same. Week numbering for intro weeks uses negative indices (-1, 0) so the main cycle starts at week 1.

### DayDefinition

Describes one training day within a week.

```
DayDefinition
├── name: String                    ("Heavy", "Upper Power", "Lower Hypertrophy")
├── shortLabel: String              ("H", "UP", "LH")
├── suggestedWeekday: Int?          (2=Mon, 4=Wed, etc. — suggestion only)
├── exerciseSlots: [ExerciseSlot]
```

### ExerciseSlot

A slot in a day for one exercise. Defines what to do and how to progress.

```
ExerciseSlot
├── role: ExerciseRole              (.primary, .accessory, .isolation)
├── defaultExercise: String         ("Bench Press")
├── alternatives: [String]          (["Incline Bench", "DB Bench"])
├── muscleGroup: String?            ("chest", "back", "legs")
├── setGroups: [SetGroup]
├── progression: ProgressionRule
```

### SetGroup

The fundamental building block. A group of sets performed together, optionally repeated. This captures DeLorme's pyramid series, straight sets, drop sets, supersets — all as the same concept.

```
SetGroup
├── sets: [SetScheme]
├── repeatCount: RepeatMode         (.fixed(4), .max, .matchDay("Heavy"))
├── restBetweenSets: Int?           (per-slot override of Program.setRestDuration)
├── restAfterGroup: Int?            (per-slot override of Program.seriesRestDuration)
```

### SetScheme

One set within a group.

```
SetScheme
├── intensity: Double               (percentage of RM: 0.5, 0.75, 1.0)
├── reps: Int                       (target reps)
├── isAMRAP: Bool                   (as many reps as possible)
├── rpe: Double?                    (rate of perceived exertion, for future use)
```

### ProgressionRule

When and how to increase weight.

```
ProgressionRule
├── trigger: ProgressionTrigger
│   ├── .completeAllSets            (hit all prescribed reps)
│   ├── .completeNGroups(n: Int)    (DeLorme: complete 5 groups)
│   ├── .amrapThreshold(reps: Int)  (nSuns: hit N+ on AMRAP)
│   └── .manual                     (user decides)
├── action: ProgressionAction
│   ├── .addWeight(kg: Double, lbs: Double)
│   └── .percentageIncrease(Double)
```

### RepeatMode

How many times a set group is repeated.

```
RepeatMode
├── .fixed(Int)                     (PHUL: 4 sets, always 4)
├── .max                            (DeLorme Heavy: as many as you can)
├── .matchDay(String)               (DeLorme Light/Medium: match Heavy's count)
```

            exerciseSlots: [
                ExerciseSlot(

                        SetGroup(

## PHUL Expressed as ProgramDefinition

```
ProgramDefinition(
    name: "PHUL",
    cycleLength: nil,
    repeating: true,
    days: [
        DayDefinition(
            name: "Upper Power", shortLabel: "UP",
            exerciseSlots: [
                ExerciseSlot(
                    role: .primary,
                    defaultExercise: "Bench Press",
                    setGroups: [
                        SetGroup(
                            sets: [SetScheme(intensity: 1.0, reps: 5)],
                            repeatCount: .fixed(4),
                        )
                    ],
                    progression: ProgressionRule(
                        trigger: .completeAllSets,
                        action: .addWeight(kg: 2.5, lbs: 5.0)
                    )
                ),
                ExerciseSlot(
                    role: .primary,
                    defaultExercise: "Barbell Row",
                    // 4x5 similar...
                ),
                ExerciseSlot(
                    role: .accessory,
                    defaultExercise: "Overhead Press",
                    setGroups: [
                        SetGroup(
                            sets: [SetScheme(intensity: 0.85, reps: 8)],
                            repeatCount: .fixed(3),
                        )
                    ],
                    // ...
                ),
                // Curls, triceps accessories...
            ]
        ),
        DayDefinition(name: "Lower Power", shortLabel: "LP", ...),
        DayDefinition(name: "Upper Hypertrophy", shortLabel: "UH", ...),
        DayDefinition(name: "Lower Hypertrophy", shortLabel: "LH", ...),
    ]
)
```

Key differences from DeLorme:
- 4 days per week instead of 3
- Repeating cycle (no fixed end)
- Straight sets (.fixed(4)) instead of pyramid series
- Mixed rep ranges (5 for power, 8-12 for hypertrophy)
- Multiple exercises per day (5-7 vs 2)
- Primary + accessory exercise roles

## Generic WorkoutEngine

Replaces DeLormeEngine. Reads any ProgramDefinition and generates workouts.

### Responsibilities

1. **Generate workout** — Given a ProgramDefinition, current day, and exercise state, produce the planned sets
2. **Calculate weights** — Apply intensity percentages to current RM values
3. **Determine next workout** — Based on completed sessions, figure out next day in the cycle
4. **Apply progression** — After a session, evaluate ProgressionRules and update RMs
5. **Handle intro cycles** — Ramp-up logic for programs that support it

### What It Replaces

| DeLormeEngine method | WorkoutEngine equivalent |
|---------------------|--------------------------|
| `intensityLevels(for:week:)` | Read from `SetGroup.sets` |
| `seriesCount(week:dayType:mondaySeriesCount:)` | Read from `SetGroup.repeatCount` |
| `generateSeries(exercises:dayType:week:)` | `generateWorkout(definition:day:week:exercises:)` |
| `calculate10RM(initial:increment:completed:)` | `applyProgression(rule:history:)` |
| `nextWorkout(program:sessions:)` | `nextDay(definition:completedDays:)` |

## Model Changes

### Program (SwiftData)

Add:
- `programType: String` — references ProgramDefinition.id
- Remove hardcoded `weekRange` — derive from ProgramDefinition.cycleLength

### Exercise (SwiftData)

No changes needed. The RM, increment, unit, and sort order fields are already generic.

### WorkoutSession (SwiftData)

Change:
- `dayType: DayType` → `dayName: String` — generic day identifier ("Heavy", "Upper Power", etc.)
- Keep `weekNumber` as-is

### CompletedSet (SwiftData)

Change:
- `seriesNumber: Int` → `groupNumber: Int` — which set group repetition
- `intensity` stays as-is (still useful for all programs)
- `targetReps` stays (already generic, just always 5 in DeLorme)

### DayType enum

Remove entirely. Day names come from ProgramDefinition.days.

### Migration

SwiftData schema migration for field renames and type changes. Since SwiftData lightweight migration doesn't support field renames automatically, use a `VersionedSchema` with a `MigrationPlan`:

1. Define `StronqSchemaV1` (current) and `StronqSchemaV2` (new field names)
2. Custom migration stage maps: `dayType` enum raw value → `dayName` string, `seriesNumber` → `groupNumber`
3. Add `programType: String` to Program with default value `"delorme-classic"` for existing data
4. Existing DeLorme sessions map: `"Heavy"/"Light"/"Medium"` → dayName, seriesNumber → groupNumber
5. `DayType` is currently `Codable` — SwiftData stores the enum's raw string value, so the migration reads those raw strings directly into the new `dayName: String` field (no data loss)
6. `SessionSummary` is not a SwiftData model (plain struct) — just update the property type, no migration needed

## UI Changes

### WorkoutSessionView

- Replace series card layout with a generic set group layout
- When a group has multiple sets (DeLorme pyramid), show them grouped
- When a group has one set repeated (PHUL straight sets), show as a flat list
- "Max groups" and "Fixed groups" replace "Max series" and "Fixed series"
- Day type label reads from dayName string instead of DayType enum

### ProgramOverviewView

- Week cards work with dynamic day count (3 for DeLorme, 4 for PHUL)
- Day dots adapt to program's days
- Arrow-up indicator uses ProgressionRule instead of hardcoded "5 series" check

### TodayView

- Day header reads from ProgramDefinition instead of DayType
- Workout preview generates from WorkoutEngine

### OnboardingFlow

- Program picker shows all available ProgramDefinitions
- Weight setup step adapts to selected program's exercises
- Config step shows program-specific options (intro cycle for DeLorme, none for PHUL)

### SettingsView

- Exercise list driven by current program's exercise slots
- "Swap exercise" reads alternatives from ExerciseSlot

## Testing Strategy

### Critical: DeLorme Parity

The refactored engine MUST produce identical workouts to v1.0 for DeLorme programs. Test plan:

1. Snapshot all DeLormeEngine outputs for every week/day combination
2. Run same inputs through WorkoutEngine with DeLorme ProgramDefinition
3. Assert identical weights, reps, set counts, progression behavior
4. Run existing DeLormeEngineTests against new engine

### PHUL Validation

- Verify correct exercises per day (Upper Power, Lower Power, Upper Hypertrophy, Lower Hypertrophy)
- Verify correct set/rep schemes per exercise role
- Verify progression triggers correctly after completing prescribed sets
- Verify repeating cycle loops back to day 1 after day 4

## Design Constraints

- **One program at a time.** User picks a program, runs it, finishes or resets before switching. No concurrent programs.
- **Single-week cadence.** Each week has the same day structure. Two-week alternating cadence is deferred to post-v2.0.
- **Backward compatible data.** Existing v1.0 workout history must survive the migration. DeLorme sessions map cleanly to the new schema.
- **DeLorme must be identical.** Users who picked DeLorme should see zero behavioral changes. Same weights, same progression, same UI flow. The refactor is invisible to them.
- **Future-proof for custom programs.** ProgramDefinition should be serializable (Codable) so users can eventually create and save their own in v2.0.
